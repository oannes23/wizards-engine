## Overview

Simple authentication and authorization for a small, fixed group. No external auth providers — this is a trusted small group. The GM creates the game, invites players, and manages access.

**Auth model**: Magic links + httpOnly cookies. Each user has a permanent login code stored in the database. The magic link URL (`/login/<code>`) sets a cookie in the browser. The API reads auth exclusively from cookies — no Bearer headers, no JWT, no OAuth.

The GM is not a distinct entity — they are a player with unlimited privileges. They have a display name (displayed as "GM [name]") and can optionally own a character via a dedicated endpoint.

No game-level settings object. The campaign is implicit — it's the only one.

## Core Concepts

### User Model

All users (GM and players) share the same base model: an ID (ULID), display name (1–50 characters, trimmed, non-empty), role (`gm`/`player`), login code (plaintext, indexed), optional linked character (`character_id`), and active status. The GM is distinguished only by their role flag — not a separate entity type.

Users also have **starring** — a list of Game Objects tracked for the starred feed. Stored in a `starred_objects` table (not on the User record). See [feed.md](feed.md).

### Magic Link Auth

Every user has a **login code** — a random opaque string stored in plaintext in the database. The magic link is a frontend URL: `/login/<code>`.

- **Login flow**: Player visits `/login/<code>`. Frontend calls `POST /api/v1/auth/login` with the code. Server validates, sets an httpOnly cookie, returns user info with `type: "user"`. Frontend redirects to app.
- **Invite flow**: If the code matches an unconsumed invite (not yet a user), the login endpoint returns `{type: "invite"}` (no cookie set). Frontend shows the join form. Player submits character name + display name via `POST /api/v1/game/join`.
- **Cookie**: httpOnly, Secure (HTTPS), SameSite=Lax. Contains the login code. Persistent — implemented as `max_age=365*24*60*60` (1 year) so the cookie survives browser restarts. Login codes themselves never expire; the 1-year max_age is the practical mechanism for persistence.
- **No Bearer headers**: API reads auth exclusively from cookies. No programmatic API auth for MVP.
- **Codes never expire**: Permanent until refreshed by the user or regenerated by the GM.
- **Plaintext storage**: Login codes are stored in plaintext (indexed for lookup). Acceptable for a trusted small group on a private server. Enables GM to view and share player links.

### Game Creation & Onboarding

1. **First-run setup**: `POST /api/v1/setup` creates the GM account (accepts `display_name`), generates a login code, sets the auth cookie, and returns the GM's magic link URL. This endpoint locks permanently after first use (checked via DB — if a GM user exists, the endpoint returns 409 Conflict). No guard — trusted network, GM starts the server.
2. **GM generates bare invite codes** via `POST /api/v1/game/invites`. The invite code is a ULID — no character is pre-linked. The response includes the magic link URL (`/login/<code>`). No expiry — valid until consumed or deleted by GM.
3. **GM shares the magic link** with the player out-of-band (text message, email, etc.).
4. **Player visits the magic link** (`/login/<code>`). Frontend calls `POST /api/v1/auth/login`. Server detects an unconsumed invite and returns `{type: "invite"}`. Frontend shows a join form.
5. **Player joins** via `POST /api/v1/game/join` with `{code, character_name, display_name}`. The system atomically: marks the invite as consumed, creates a full Character (all mechanical fields defaulting to 0), creates a User account (login code = the invite code), and links them. Sets the auth cookie. Returns user info.
6. **Same magic link now works as permanent login**: subsequent visits to `/login/<code>` set the cookie and redirect to the app. The invite code became the user's login code.

### Link Refresh

Any authenticated user can refresh their own magic link via `POST /api/v1/me/refresh-link`. This generates a new login code, invalidates the old one, updates the cookie, and returns the new magic link URL. The old link stops working immediately.

The GM can also regenerate any player's login code via `POST /api/v1/players/{id}/regenerate-token`. This returns the new magic link URL for the GM to share with the player.

### Auth Errors

Standard HTTP status codes plus a JSON error body:

| Status | Meaning | Example `code` values |
|--------|---------|----------------------|
| `401` | Missing or invalid cookie | `cookie_missing`, `cookie_invalid`, `account_inactive` |
| `403` | Valid cookie but insufficient permissions | `insufficient_role`, `not_owner` |

Error response body:
```json
{
  "error": {
    "code": "cookie_invalid",
    "message": "The provided auth cookie is not valid."
  }
}
```

Login/join-specific errors:
- `POST /api/v1/auth/login`: `404` with code `code_not_found` for invalid, consumed-but-unlinked, or nonexistent codes.
- `POST /api/v1/game/join`: `404` with code `invite_not_found` for invalid, already-consumed, or nonexistent invite codes. Don't leak whether a code exists.

### Permission Model

Three roles:

**GM (Game Master)**:
- Full read/write access to everything
- Can create/modify all game objects (NPCs, Groups, Locations, Stories, Sessions)
- Can approve/reject proposals (including their own — same workflow as players)
- Can directly modify any character or game state
- Can manage sessions (create, start, end)
- Can generate and delete invite codes
- Can regenerate any player's login code (returns new magic link)
- Can view all player magic links via the player roster (including viewer users)
- Can change any event's visibility level
- Can access the **silent feed** (`/me/feed/silent`) — bookkeeping events invisible to players and viewers
- Can override Story visibility (add player IDs to `visibility_overrides`)
- Can optionally own a character via `POST /api/v1/me/character`
- Sees all events regardless of bond-distance visibility (unified visibility model — see [feed.md](feed.md))
- `GET /me` returns `can_view_gm_content: true, can_take_gm_actions: true`

**Players**:
- Can read all public game state (characters, groups, locations, NPCs, stories, sessions)
- Can read the player roster (`/api/v1/players`) — names, roles, character links (no login URLs); viewers are excluded from the roster for player callers
- Can read events and story entries filtered by unified visibility model (bond-graph based, server-side) — see [feed.md](feed.md)
- Can access personal feed (`/me/feed`), starred feed (`/me/feed/starred`), and per-Game Object feeds
- Can star/unstar Game Objects (`/me/starred`)
- Can modify their own character (name, description, notes, direct actions)
- Can update their own display name via `PATCH /api/v1/me`
- Can refresh their own magic link via `POST /api/v1/me/refresh-link`
- Can submit and revise proposals for their own character
- Can add entries to Stories they can see (see = write)
- Cannot modify other players' characters
- Cannot access GM-only endpoints (including silent feed)
- `GET /me` returns `can_view_gm_content: false, can_take_gm_actions: false`

**Viewers**:
- GM-level read access: sees all events (except `silent`), all proposals, the GM dashboard, queue summary, all characters' effects, and the full player roster (without login URLs)
- Cannot take any mutating actions — all POST/PATCH/DELETE endpoints return 403
- Cannot access the silent feed (`/me/feed/silent`)
- Cannot submit proposals, approve/reject proposals, or use any player direct actions
- Can star/unstar Game Objects (`/me/starred`) — personal convenience feature with no game-state impact
- Has no linked character — viewer accounts are created via viewer invites
- Visible to GM and viewer callers in the player roster; hidden from player callers' roster
- `GET /me` returns `can_view_gm_content: true, can_take_gm_actions: false`

### Invite & Character Lifecycle

- One invite code = one future user account. Invites carry a `role` field (`"player"` or `"viewer"`, default `"player"`). Invites are **bare** — not pre-linked to a character.
- Invite codes are ULIDs — single-use, consumed on redemption, no expiry.
- On player redemption: the invite code becomes the user's login code. PC + User created atomically.
- On viewer redemption: User created with no character. `character_name` field is not required in the join request.
- GM can delete unconsumed invite codes.
- If a player needs a new character (death, replacement): GM issues a new invite. Old player account is deactivated. Player joins with the new invite link.
- Players always have exactly one character. Viewers never have a character.
- When a player account is deactivated, their pending proposals are **orphaned in place** — the GM can still approve/reject them. The character persists (soft delete is optional and separate from account deactivation).

### GM Self-Play

The GM can optionally own a character via `POST /api/v1/me/character`, which accepts `{name}` and creates a full Character (all mechanical fields defaulting to 0) linked to the GM account. If the GM already has a linked character, the old character remains in the system as an ownerless full Character, and the new one is created and linked. The GM's character uses the same proposal workflow — the GM must explicitly approve their own proposals.

## Decisions

### No External Auth

- **Decision**: No OAuth, no external identity providers. Magic links with cookies.
- **Rationale**: This is a trusted small group (4–6 players + 1 GM). The complexity of external auth is not justified.
- **Implications**: Auth management is simple. Acceptable for the use case.

### Magic Link Authentication

- **Decision**: Auth via permanent magic link URLs (`/login/<code>`) that set httpOnly cookies. API reads auth exclusively from cookies. No Bearer headers.
- **Rationale**: Magic links eliminate password management entirely. Players bookmark their link or re-request it from the GM. Cookies are the natural browser auth mechanism. A trusted small group doesn't need programmatic API access for MVP.
- **Implications**: Frontend must handle the magic link → login/join flow. Server sets cookies on login, join, setup, and link refresh. All API middleware reads from cookie, not Authorization header.

### Plaintext Login Code Storage

- **Decision**: Login codes stored in plaintext in the database (indexed for lookup).
- **Rationale**: Enables the GM to view and share any player's magic link without regenerating it. For a trusted small group on a private server, the DB-leak threat model is low. Hashing would prevent the "GM can see all links" feature.
- **Implications**: If the database is compromised, all login codes are exposed. Acceptable tradeoff for this use case.
- **Alternatives considered**: SHA-256 hashing (prevents GM viewing links without regen), reversible encryption (complex for no real benefit at this scale).

### Cookie-Only API Auth

- **Decision**: API reads auth exclusively from cookies. No Bearer header support.
- **Rationale**: The only client is a browser-based web UI. Cookies are set automatically on every request. No need for programmatic API access for MVP.
- **Implications**: Non-browser clients (scripts, bots) cannot authenticate. Can be added later if needed.

### Roles

- **Decision**: Three roles — GM, Player, and Viewer.
- **Rationale**: The GM has full read/write access. Players interact through characters. Viewers have GM-level read access but cannot take any actions — they are read-only spectators. This enables trusted observers (friends, co-GMs, note-takers) to follow the campaign without needing GM credentials.
- **Implications**: Auth dependency layer distinguishes between privileged read access (GM + Viewer) and mutation rights (GM only). Viewer accounts are created via invites with a role field.
- **Alternatives considered**: Sharing GM credentials (security risk, no audit trail), screen sharing (no independent access).

### GM as Privileged Player

- **Decision**: The GM is a player with unlimited privileges, not a distinct entity type. Has a display name (displayed as "GM [name]"). Can optionally own and play a character via `POST /api/v1/me/character`.
- **Rationale**: In many TTRPG groups, the GM also plays a character (GMPC). Treating the GM as a special player simplifies the user model.
- **Implications**: Single user model for all accounts. GM's character uses the same proposal workflow as any player — GM must explicitly approve their own proposals.

### GM Character Endpoint

- **Decision**: `POST /api/v1/me/character` (GM only) creates a full Character and links it to the GM account. Accepts `{name}`. If the GM already has a character, the old one stays as an ownerless full Character.
- **Rationale**: The GM can't use the invite flow (they already have an account). A dedicated endpoint is cleaner than overloading PATCH /me with character creation logic.
- **Implications**: Ownerless full Characters are a valid state (GM's previous characters). These characters have full detail but no player controlling them.

### Bare Invite Flow

- **Decision**: Invites are bare — not pre-linked to a character. The invite code (ULID) becomes the user's permanent login code on redemption. `POST /api/v1/game/join` accepts `{code, character_name, display_name}` and atomically creates User + Character.
- **Rationale**: Decouples invite generation from character creation. The player names their own character. Simpler invite table (no character_id at creation). The invite code doubling as the login code eliminates a separate token generation step.
- **Implications**: `invites` table: `id` (ULID, this IS the code), `is_consumed`, `created_at`. `character_id` is set on redemption (nullable). No character pre-creation required before inviting.

### Login Code Format

- **Decision**: ULID for initial codes (from invites). New codes generated on refresh/regen use `secrets.token_urlsafe(32)` (43-character URL-safe string).
- **Rationale**: Invite codes are ULIDs for consistency with the rest of the system. Refreshed codes don't need to be ULIDs — URL-safe random strings provide sufficient entropy and are shorter.
- **Implications**: Login code column must accommodate both formats (TEXT type, no fixed length constraint).

### No Login Code Expiry

- **Decision**: Login codes never expire. Permanent until refreshed by the user or regenerated by the GM.
- **Rationale**: Simplest approach for a trusted small group. No refresh token complexity.
- **Implications**: GM regeneration and player self-refresh are the safety valves for compromised codes.

### Login Response Discriminator

- **Decision**: `POST /api/v1/auth/login` always returns a `type` field: `"user"` when the code resolves to an active user account, `"invite"` when it resolves to an unconsumed invite.
- **Rationale**: The two response shapes carry different fields (user response includes `id`, `display_name`, `role`, `character_id`; invite response carries only `type`). A discriminator field lets the frontend branch without inspecting which fields are present.
- **Implications**: Both `LoginUserResponse` and `LoginInviteResponse` Pydantic models include `type` as a literal field with a fixed default value. Inactive users with a matching login code fall through to 404 to avoid leaking account-state information.

### Player Self-Refresh

- **Decision**: Any authenticated user can refresh their own magic link via `POST /api/v1/me/refresh-link`. Generates a new code, invalidates the old link, updates the cookie, returns the new magic link URL.
- **Rationale**: Players should be able to invalidate their own link if they accidentally share it or suspect compromise. Empowers users without requiring GM intervention.
- **Implications**: Old link stops working immediately. If the user has the site open in other browsers/devices, those sessions lose auth and must re-login via the new link.

### Logout Without Code Invalidation

- **Decision**: `POST /api/v1/auth/logout` clears the httpOnly cookie but does not rotate or invalidate the login code. The magic link remains usable. Deliberately unauthenticated — clearing a non-existent cookie is harmless, and requiring auth on logout creates a catch-22 when the cookie is stale.
- **Rationale**: Logout is a browser-session action, not a credential rotation. Users who want a secure logout (e.g., on a shared device) should call `POST /me/refresh-link` first to rotate the code.
- **Implications**: Cookie-clearing logout alone does not protect against a stolen magic link. Self-refresh is the correct path for credential invalidation.

### Player-Character Mapping

- **Decision**: Strictly one-to-one. Each player owns exactly one character.
- **Rationale**: Simplifies ownership, proposals, event visibility, and all per-character logic.
- **Implications**: Character reassignment requires a new invite (old account deactivated).

### Player Display Name Updates

- **Decision**: Players can update their own display name via `PATCH /api/v1/me`. GM cannot change player display names.
- **Rationale**: Players should control their own identity. GM doesn't need to manage player names for a trusted group.
- **Implications**: `PATCH /api/v1/me` accepts `{display_name}` (1–50 chars, trimmed, non-empty). Same validation as initial setup.

### Token Regeneration

- **Decision**: GM can regenerate any player's login code via `POST /api/v1/players/{id}/regenerate-token`. Returns the new magic link URL. Old link stops working immediately. GM can also regenerate their own code.
- **Rationale**: Acts as both a kick (don't share new link) and a password reset (share new link). Simpler than a separate ban/unban system.
- **Implications**: Player must obtain the new magic link from the GM to regain access.

### No Explicit Deactivation Endpoint

- **Decision**: No dedicated deactivation/ban endpoint. Token regeneration is the "soft ban" (don't share new link). Account deactivation only happens as a side effect of re-inviting (new invite for new character → old account deactivated).
- **Rationale**: For a trusted small group of 4–6 players, a dedicated ban system is over-engineering. Token regen achieves the same effect.
- **Implications**: `is_active` is only set to `false` during the re-invite flow, not via a standalone API call.

### First-Run Setup

- **Decision**: One-time `POST /api/v1/setup` endpoint. Locked by checking if a GM user already exists in the DB (returns 409 Conflict if so). No guard — first request wins. Sets the auth cookie and returns the GM's magic link URL.
- **Rationale**: Server runs on a trusted local network. GM starts the server and immediately hits setup. No risk of hijacking. DB check is the simplest permanent lock mechanism.
- **Implications**: No separate lock file or config needed. Server should log clearly when setup is completed. GM must save their magic link immediately.

### Player Profile

- **Decision**: Display name only (1–50 characters, trimmed whitespace, non-empty after trim). No avatar, no color, no preferences.
- **Rationale**: Minimal surface area. Frontend can derive colors from player ID if needed. Trusted group — no need for strict name validation.
- **Implications**: `display_name` is the only user-editable profile field (via `PATCH /api/v1/me`).

### No Game Settings Object

- **Decision**: No `GET/PATCH /api/v1/game` endpoints for MVP. The campaign is implicit — it's the only one. Player roster is served via a dedicated `/api/v1/players` endpoint.
- **Rationale**: A game settings object adds a model and endpoints for no clear MVP value. Everything game-level (name, config) can be added later if needed.
- **Implications**: No `game` table. Player roster is a separate concern.

### Player Roster Visibility

- **Decision**: All authenticated users can see the player roster via `GET /api/v1/players`. Returns display_name, role, character_id, is_active for all users. For GM callers, also includes `login_url` for each player (omitted for non-GM callers).
- **Rationale**: Players need to know who's in the game for the UI. GM needs to be able to view and share player magic links. No sensitive data in the base roster.
- **Implications**: Single endpoint with role-conditional response fields.

### Auth Error Responses

- **Decision**: Standard HTTP 401 (missing/invalid cookie) and 403 (insufficient role) plus a JSON error body with a machine-readable `code` field.
- **Rationale**: Clear distinction between "who are you?" (401) and "you can't do that" (403) helps client UX. Machine-readable codes enable client-side error handling without parsing messages.
- **Implications**: Consistent error format across all endpoints. Error codes are documented and stable.

### Deactivation Leaves Proposals Orphaned

- **Decision**: When a player account is deactivated, their pending proposals remain in place. The GM can still approve/reject them.
- **Rationale**: The GM should decide what to do with pending proposals. The character still exists — deactivation only affects login access. Auto-rejection would be lossy.
- **Implications**: No cascade logic on deactivation. Pending proposals for inactive characters are a valid state.

### Unified Visibility Filtering

- **Decision**: Server-side filtering for players using the unified 7-level visibility model (see [feed.md](feed.md)): `silent`, `gm_only`, `private`, `bonded`, `familiar`, `public`, `global`. Computed on read via Character-intermediary bond-graph traversal. GM sees all events (except `silent` — use silent feed). GM can permanently change any individual event's visibility level and override Story visibility.
- **Rationale**: Server-side filtering is the only secure approach. Compute-on-read is sufficient for 4–6 players. Per-event override is simpler than per-player grants.
- **Implications**: Feed and event endpoints must compute bond-graph proximity for the requesting player's character. GM override modifies the event's stored visibility level or the Story's `visibility_overrides`.

### GM Self-Play Proposals

- **Decision**: GM submits proposals for their own character using the same workflow as players. Must explicitly approve their own proposals.
- **Rationale**: Maintains a clean audit trail. The approval step is lightweight for the GM but ensures all state changes are logged consistently.
- **Implications**: No special-casing in the proposal system for GM-owned characters.

### Identity Endpoint

- **Decision**: `GET /api/v1/me` returns the current user's identity: player ID, display name, role, linked character ID.
- **Rationale**: Clients need a bootstrap endpoint to determine who the current user is and what character they control.
- **Implications**: Simple read-only endpoint. No auth-specific data returned (no login code echo).

### Invite Error Response

- **Decision**: `POST /api/v1/game/join` returns `404 Not Found` with code `invite_not_found` for all invalid invite attempts — consumed, missing, or malformed. Don't leak whether a code exists.
- **Rationale**: Prevents enumeration of invite codes. For a trusted group the security benefit is marginal, but it's also the simplest implementation (one error path).
- **Implications**: Client cannot distinguish between "code doesn't exist" and "code already used." This is intentional.

## API Endpoints

### Auth & Onboarding

| Method | Endpoint | Access | Purpose |
|--------|----------|--------|---------|
| `POST` | `/api/v1/setup` | Unauthenticated (one-time) | Create GM account (accepts `{display_name}`). Sets auth cookie. Returns magic link URL. Returns 409 if already set up. |
| `POST` | `/api/v1/auth/login` | Unauthenticated | Validate a login code `{code}`. If code matches an active user → sets cookie, returns user info with `type: "user"`. If code matches an unconsumed invite → returns `{type: "invite"}` (no cookie). Otherwise → 404. Inactive users with a matching code fall through to 404 (no account-state leak). |
| `POST` | `/api/v1/game/join` | Unauthenticated | Redeem invite code `{code, display_name, character_name?}`. For player invites: `character_name` required — creates User + Character atomically. For viewer invites: `character_name` omitted — creates User only (no character). Sets auth cookie. Returns user info including `role` and nullable `character_id`. 404 if code is invalid/consumed. |
| `GET` | `/api/v1/me` | Any authenticated | Current user identity (ID, name, role, character ID) |
| `PATCH` | `/api/v1/me` | Any authenticated | Update own display name `{display_name}` |
| `POST` | `/api/v1/me/refresh-link` | Any authenticated | Refresh own magic link. Returns new magic link URL. Updates cookie. Old link stops working. |
| `POST` | `/api/v1/auth/logout` | Unauthenticated | Clear the httpOnly auth cookie. Returns 204. Does not invalidate the login code — the magic link remains usable. |
| `POST` | `/api/v1/me/character` | GM only | Create a full Character `{name}` and link to GM account. If GM already has a character, old one stays as ownerless. |

### Invite Management

| Method | Endpoint | Access | Purpose |
|--------|----------|--------|---------|
| `POST` | `/api/v1/game/invites` | GM | Generate bare invite code (ULID). Optional body `{role: "player"|"viewer"}` (default `"player"`). Returns invite including `role` field + magic link URL. |
| `GET` | `/api/v1/game/invites` | GM or Viewer | List invite codes (consumed and unconsumed). Response includes `role` per invite. |
| `DELETE` | `/api/v1/game/invites/{id}` | GM | Delete an unconsumed invite code |

### Player Management

| Method | Endpoint | Access | Purpose |
|--------|----------|--------|---------|
| `GET` | `/api/v1/players` | Any authenticated | Player roster (display_name, role, character_id, is_active). GM callers see all users including viewers, with `login_url` per user. Viewer callers see all users without `login_url`. Player callers see only GM + player users (viewers filtered out), without `login_url`. |
| `POST` | `/api/v1/players/{id}/regenerate-token` | GM | Regenerate a player's login code. Returns new magic link URL. Old link stops working. |

### Feed & Starring (defined in [feed.md](feed.md), auth-relevant)

| Method | Endpoint | Access | Purpose |
|--------|----------|--------|---------|
| `GET` | `/api/v1/me/feed` | Any authenticated | Complete personal feed (visibility-filtered) |
| `GET` | `/api/v1/me/feed/starred` | Any authenticated | Starred feed (filtered to starred objects) |
| `GET` | `/api/v1/me/feed/silent` | GM only | Silent feed (bookkeeping events) |
| `GET` | `/api/v1/me/starred` | Any authenticated | List starred Game Objects |
| `POST` | `/api/v1/me/starred` | Any authenticated | Star a Game Object `{type, id}` |
| `DELETE` | `/api/v1/me/starred/{type}/{id}` | Any authenticated | Unstar a Game Object |
